1. Configuration: The Foundation of the Queue

The queue system's behavior is primarily controlled by three configuration variables found in `config.py`:

*   `QUEUE_ALL`: This sets the maximum number of tasks that can be active (both downloading and uploading) at a time.
*   `QUEUE_DOWNLOAD`: This sets the maximum number of concurrent downloads.
*   `QUEUE_UPLOAD`: This sets the maximum number of concurrent uploads.

The default value for all these variables is `0`, which means the queue system is **disabled** by default. To enable it, a admin must set these variables or edit bot settings from bot to a positive integer in their values.

2. The Leech Command: A Practical Example

Let's walk through what happens when a user issues a `/leech` command.

File: `echobotz/plugins/leech.py`

When a user sends a `/leech` command, the `leech` function in this file is triggered. This function creates a `Mirror` object with `is_leech=True` and calls the `new_event` method. This method is the entry point for handling the leech task.

3. Pre-Task Checks: To Queue or Not to Queue?

File: `echobotz/helper/ext_utils/task_manager.py`

Before the download even begins, the `new_event` method in `leech.py` calls the `pre_task_check` function from `task_manager.py`. This function performs several checks, including:

*   **User authorization:** It verifies if the user is authorized to use the bot.
*   **Task limits:** It checks if the user has exceeded their maximum concurrent tasks.
*   **Bot limits:** It checks if the bot has reached its global maximum task limit.

If any of these checks fail, the task is rejected. If they pass, the process continues to the queuing logic.

4. The Queuing Logic: Where the Magic Happens

**File:** `echobotz/helper/ext_utils/task_manager.py`

The core of the queueing system resides in the `check_running_tasks` function. This function is called after the `pre_task_check`. Here's how it works:

1.  **Checking Limits:** The function checks the current number of active downloads and uploads against the limits defined in `QUEUE_ALL`, `QUEUE_DOWNLOAD`, and `QUEUE_UPLOAD`.
2.  **Adding to the Queue:** If the number of active tasks has reached the configured limits, the task is added to the appropriate queue:
    *   If the download limit is reached, the task is added to the `queued_dl` dictionary.
    *   If the upload limit is reached, the task is added to the `queued_up` dictionary.
3.  **Event Object:** When a task is queued, an `asyncio.Event` object is created and associated with the task. The task will then wait until this event is set.
4.  **No Queueing:** If the limits have not been reached, the task is added to the `non_queued_dl` or `non_queued_up` set, and the download/upload process begins immediately.

#### 5. The Download and Upload Queues

The bot maintains two separate queues:

*   `queued_dl`: For tasks waiting to be downloaded.
*   `queued_up`: For tasks that have finished downloading and are waiting to be uploaded.

This separation allows the bot to manage download and upload resources independently. For a leech task, it will first go through the download queue (if necessary), and after the download is complete, it will go through the upload queue (if necessary).

6. Processing the Queue: The `start_from_queued` Function

**File:** `echobotz/helper/ext_utils/task_manager.py`

The `start_from_queued` function is responsible for processing the queues. This function is called whenever a task completes. It checks the `queued_dl` and `queued_up` dictionaries and, if there are any waiting tasks and the limits allow, it will:

1.  **Select a Task:** It picks the next task from the queue.
2.  **Set the Event:** It calls the `.set()` method on the `asyncio.Event` object associated with the task.
3.  **Start the Task:** This triggers the waiting task to resume its execution (either downloading or uploading).
